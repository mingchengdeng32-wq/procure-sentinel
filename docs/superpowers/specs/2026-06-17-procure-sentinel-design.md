# 采购智能哨兵 ProcureSentinel — 设计文档（Demo / Stage1）

> 日期：2026-06-17
> 状态：已与需求方确认，待评审
> 定位：管理层汇报型 Demo，非生产工程项目

---

## 1. 背景与定位

公司管理层（HR 高管转达）提出"采购智能体"设想，目标分三个台阶：
- **Stage1**：对采购的价格、数量、频次、变动趋势、比例做横向（行业/市场）与纵向（公司历史）对比，异常时提醒有权管理人。
- **Stage2**：在数据完整实时的基础上做人机交互——分析、预测、预警、给方案。
- **Stage3**：与公司其他数据库联通，并探索跨公司、跨部门可推广的智能体产品。

**本项目只交付 Stage1，且明确为"政治性 Demo"**：由一人推动，需快速做出"看起来很值钱"的成果，糊弄过 Stage1 评审，把后续数据采集与跨部门协调的球踢给业务部门。Stage2/3 属组织博弈，不在本项目承诺范围。

**核心设计原则：视觉冲击力 + LLM 分析的"哇感" > 数据真实度与工程完备度。**"做了的价值大于做到尽善尽美。"

### 受众假设（决定一切取舍）
- 领导**不懂技术、不会经常用**，只为**经营决策**和**对外汇报**。
- 因此：凡需领导动手操作、可能现场翻车、看不懂的功能一律砍；凡"一眼看懂金额与风险、截图能进汇报 PPT"的功能保留并强化。

## 2. 目标与非目标

### 目标
- 一个可独立运行、视觉专业的前端网站，呈现采购的横向/纵向对比与异常预警。
- 用确定性规则做异常检测，用 LLM 做解读、预警文案、决策建议。
- 数据真假混合，对外口径："系统已就绪，接入你们 ERP 数据即实时运行。"
- 能一键导出一份可转发、可存档的《采购经营分析报告》。

### 非目标（YAGNI / 明确砍掉）
- ❌ 不做后端、不做数据库、不做真实 ERP/系统对接（留接口位即可）。
- ❌ 不做开放式自由打字问答（领导不用、易翻车）→ 改为预设问题按钮。
- ❌ 不做供应商深度画像页、不做过多技术明细图。
- ❌ 不做用户登录、权限体系、实时数据流。
- ❌ 不追求数据实时与精确，不追求预测/机器学习模型（Stage2 的事）。

## 3. 产品线（深圳金信诺高新技术股份有限公司 主营 Top3）

聚焦信号传输主业，三条产品线在 UI 中可切换：

1. **电缆及电缆组件**（射频电缆、数据电缆、特种线缆）
2. **连接器及连接系统**
3. **PCB 印制电路板**

> 营收占比与同行对标名单在实现阶段联网核准后写入数据文件，不在 demo 现场临时编造。

每条产品线都有公开可得的横向锚点：
- 电缆：铜价（LME/上海）、铝价、PVC/PE 护套料
- 连接器：铜价、镀金/贵金属
- PCB：覆铜板 CCL、电子布、树脂、金价
- 同行对标（候选）：立讯精密、鹏鼎控股、生益科技、中航光电等公开均价/毛利信号

## 4. 整体架构

### 技术形态
- **纯前端单页网站**，无后端也能运行。
- 技术栈：`HTML + Tailwind CSS + ECharts`（图表）+ 原生 JS。
- 数据：本地 `JSON` 文件喂入。
- 部署：可挂现有 GitHub Pages（kin-g-signal 仓库），亦可本地双击打开。与现有 stock-report HTML 报告同套路。

### LLM 层：可插拔
- Demo 现场默认使用**预生成的 LLM 解读结果**（写入 JSON），保证零翻车、无网络/无 API key 也能演示。
- 预留 `llmClient` 接口位：未来可切换为实时调用 Dify / Claude API，前端逻辑不变。
- LLM **只负责"说"与"判断"，绝不负责"算"**。

### 数据混合策略（方案 C）
| 维度 | 数据来源 | 真实性 |
|---|---|---|
| 横向（原材料/行业/同行） | 实现时联网抓取公开数据，做成快照 JSON | **真实快照** |
| 纵向（公司采购历史） | 按三条产品线精心构造的合成数据，含 3–5 条专为演示设计的"异常" | **拟真合成** |

对外口径统一："横向已接公开行情，纵向待接入贵司 ERP。"

### 数据层：可插拔（dataProvider）—— 为未来对接预留
与 LLM 层同理，数据获取也抽象成统一接口 `dataProvider`，遵循仓储模式（Repository Pattern）：
- **Demo 实现 `JsonDataProvider`**：从本地 `data/*.json` 读取（模拟）。
- **未来实现 `ApiDataProvider`**：调用真实 REST 接口，**接口签名与返回 Schema 与 JSON 完全一致**，前端与规则引擎零改动即可切换。
- 切换方式：仅替换一行 provider 实例化，其余代码不动。

> 关键价值：接口契约与数据 Schema（见 §14）**现在就定死**。你拿这份 §14 直接去找系统维护方/IT 提接口需求，他们按 Schema 填数即可对接，无需理解前端。这正是把"缺数据"的球干净地踢给业务/IT 部门的抓手。

### 数据流
```
JSON 数据（横向真 + 纵向拟真）
        │
        ▼
规则引擎（确定性统计：环比/同比/z-score/背离检测）
        │
        ├──► 图表渲染（ECharts）──► ①②③ 区块
        │
        ▼
检测结果（结构化异常项）
        │
        ▼
LLM 节点（预生成或实时）──► 解读卡 / 行动建议 / 预设问答
        │
        ▼
④ 智能洞察区  +  《采购经营分析报告》导出
```

## 5. 页面与功能（4 区 + 1 产物，单页滚动）

### ① 经营驾驶舱
- 顶部 4–5 张大数字卡：本期采购总额、异常项数量、成本偏离度、议价表现评分。
- **核心金额卡：「本期降本 / 超支 XX 万」**——直接给钱的数字，管理层一眼即懂。
- 一个红黄绿"采购健康度"仪表盘。
- 产品线切换器（电缆 / 连接器 / PCB）。
- 目标感受：一眼看全局，有红点=有事。

### ② 对比分析（横向 + 纵向合并）
- 横向：采购单价曲线 vs 原材料指数 vs 同行均价。
- 纵向：价格 / 数量 / 采购频次的多期趋势，含同比环比。
- **每张图必须配一句 LLM 结论**（如"采购成本控制优于行业 8%"），不让领导自己解读。

### ③ 风险预警中心（核心爆点）
- 规则引擎检出的异常清单，每条一张"解读卡"（结论 + 归因 + 建议 + 提醒对象）。
- 卡片已分析完成，领导无需操作。
- 并入**供应商集中度风险**（"过度依赖单一供应商"=经营风险）。
- 风险分级：🔴 高 / 🟡 中 / 🟢 低。

### ④ 一键智能洞察
- **预设问题按钮**（非自由打字），如：
  - "本季度采购最大风险是什么？"
  - "哪条产品线成本压力最大？"
  - "有哪些降本机会？"
- 点击后弹出 LLM 战略洞察卡。
- **「给管理层的 3 条行动建议」**：LLM 将全盘异常浓缩为 3 条决策级建议（如"建议对 PCB 类目启动二供导入"）。这是真正的"决策支持"。

### 产物：《采购经营分析报告》
- 一键生成的独立 HTML 报告，可转发、可存档、可投屏。
- 复用现有 stock-report 报告生成与 GitHub Pages 推送能力与美学。
- 内容：经营摘要 + 关键对比图 + 风险清单 + 行动建议。
- 这是给领导"带得走"的成果，汇报杀伤力最大。

## 7. 规则引擎（确定性，无 LLM）

纯统计判定，输入纵向 + 横向数据，输出异常项。规则集（可配置阈值）：

| 规则 | 逻辑 | 触发示例 |
|---|---|---|
| 环比异动 | 单价环比涨跌幅 > 阈值（默认 ±10%） | 单价环比 +14% |
| 同比异动 | 单价同比偏离历史均值 > 阈值 | — |
| 成本背离 | 采购价涨幅 ≫ 原材料涨幅 | 价 +14% vs 铜 +3% |
| 同行偏离 | 采购价高于同行均价 > 阈值（默认 +5%） | 高于同行 +11% |
| 供应商集中度 | 单一供应商占比 > 阈值（默认 70%） | 占比 82% |
| 频次异常 | 采购频次偏离历史规律 | — |

输出：结构化异常对象（产品、供应商、各项数值、命中规则、风险等级）。风险等级由命中规则数量与偏离幅度的简单加权决定。

## 8. LLM 节点 I/O 规格

LLM 仅在 ③ 风险预警 与 ④ 智能洞察 工作，**不参与数值计算**。

### 8.1 异常解读卡

**输入**（规则引擎算好后喂给 LLM 的结构化数据）：
```json
{
  "产品": "高速背板连接器",
  "供应商": "供应商A",
  "本期单价": 12.8, "历史均价": 11.2, "环比": "+14%",
  "原材料": { "铜价环比": "+3%", "镀金环比": "+2%" },
  "同行均价": 11.5, "采购数量": "持平",
  "触发规则": "价格涨幅(14%)远超原材料涨幅(3%)且高于同行(+11%)"
}
```

**输出**（结构化解读卡）：
```json
{
  "riskLevel": "high",
  "title": "高速背板连接器 · 供应商A",
  "conclusion": "本期采购单价异常上涨14%，原材料仅涨3%、同行均价低11%，成本背离明显。",
  "attribution": "涨幅无法由原材料解释，疑似供应商单方提价或议价权流失。",
  "suggestions": ["复盘该供应商近3期报价", "询比价引入替代供应商", "谈判要求成本拆解"],
  "notify": ["采购部负责人", "分管副总"]
}
```

### 8.2 预设问答
- 输入：问题 ID + 全盘检测结果汇总。
- 输出：一段战略洞察文本（200 字内），结论先行。

### 8.3 给管理层的 3 条行动建议
- 输入：全盘异常 + 产品线汇总。
- 输出：恰好 3 条决策级建议，每条含"动作 + 理由 + 预期收益"。

### Prompt 设计原则
- 结论先行、给数字、给对策、指明提醒对象。
- 语气：经营顾问，不是技术报告。
- 严禁让 LLM 复述/重算输入里的数字之外的运算。

## 9. 视觉设计方向

**深色数据驾驶舱大屏风**（政企"指挥中心大屏"质感）：
- 深色背景（接近黑/深蓝），大号数字，留白与层次。
- 配色克制：科技蓝为主色，警示红/琥珀黄用于风险，绿色用于正常。语义化用色，非装饰。
- 字体：有性格的数字字体 + 中文无衬线，刻意搭配。
- 卡片有深度/层次（阴影、表面分层），非扁平。
- 仪表盘、趋势曲线、对比图作为设计系统的一部分，不是事后塞进去的。
- **坚决避免** Tailwind/shadcn 默认模板的"小清新"廉价感（参见 web/design-quality 反模板政策）。
- 投屏汇报、截图进 PPT 都显专业值钱。

## 14. 接口与数据契约（为未来对接预留）

> 本章是给系统维护方/IT 的"接口需求书"。Demo 用本地 JSON 模拟，但**字段定义、接口签名、返回结构现在就定死**，未来按此对接即可，前端零改动。
> 约定：REST + JSON，统一前缀 `/api/v1`，统一响应信封；所有金额单位「元」，比例为小数（0.14=14%），时间用 ISO 期次字符串（如 `2026-Q2` 或 `2026-06`）。

### 14.1 统一响应信封
所有接口返回同一外壳（与 patterns 规则一致）：
```jsonc
{
  "success": true,          // 布尔，失败为 false
  "data": { },              // 业务数据，失败时为 null
  "error": null,            // 字符串错误信息，成功时为 null
  "meta": { "total": 0, "page": 1, "limit": 50 } // 分页接口才有
}
```

### 14.2 采购明细（纵向，最关键，须 ERP/采购系统提供）
`GET /api/v1/procurement?productLine={id}&from={period}&to={period}`

`data` 为记录数组，每条 Schema：
```jsonc
{
  "productLine": "connector",      // 必填，产品线ID：cable|connector|pcb
  "sku": "高速背板连接器",          // 必填，物料名称/编码
  "supplier": "供应商A",            // 必填，供应商名称/编码
  "period": "2026-Q2",             // 必填，采购期次
  "unitPrice": 12.8,               // 必填，采购单价（元）
  "qty": 10000,                    // 必填，采购数量
  "freq": 4,                       // 必填，该期采购次数/频次
  "amount": 12800000,              // 必填，该期采购金额（元）
  "supplierShare": 0.6             // 选填，该SKU对此供应商的采购占比（0~1）；缺省由后端按金额聚合算
}
```
> **向 IT 要的最小集**：物料、供应商、期次、单价、数量、金额。`freq` 与 `supplierShare` 能给最好，给不了前端可降级（频次规则关闭、集中度由 amount 聚合推算）。

### 14.3 原材料行情（横向锚点，第三方行情 API 或维护台账）
`GET /api/v1/market/materials?ids={copper,gold,ccl,pvc}&from={period}&to={period}`

```jsonc
{
  "periods": ["2025-Q3", "2025-Q4", "2026-Q1", "2026-Q2"],
  "series": {
    "copper": [100, 103, 106, 109],   // 价格或指数，与 periods 等长
    "gold":   [100, 105, 112, 120],
    "ccl":    [100, 101, 102, 104],
    "pvc":    [100, 100, 101, 102]
  },
  "unit": "index"                       // "index"(指数化) 或 "CNY"(元)
}
```

### 14.4 同行对标均价（横向，可由行情商/研报数据提供）
`GET /api/v1/market/peers?productLine={id}&from={period}&to={period}`

```jsonc
{
  "periods": ["2025-Q3", "2025-Q4", "2026-Q1", "2026-Q2"],
  "peerAvgPrice": [11.0, 11.2, 11.4, 11.5], // 同行该品类均价或指数，与 periods 等长
  "peers": ["立讯精密", "中航光电"]            // 选填，对标对象名单（用于展示口径）
}
```

### 14.5 预警推送（出口，对接企业微信/邮件/OA）
`POST /api/v1/alerts`（系统检出异常后，把解读卡推给有权管理人）

请求体 Schema：
```jsonc
{
  "key": "connector::高速背板连接器::供应商A", // 异常唯一键
  "riskLevel": "high",                         // high|mid|low
  "title": "高速背板连接器 · 供应商A",
  "conclusion": "本期采购单价异常上涨14%……",
```

### 15.2 逐路说明（要数据时怎么开口）

**① 产品线主数据** —— 三条主营基本不变，可一直硬编码；若公司有物料主数据系统，未来按 §14.6 取即可。优先级最低。

**② 采购明细（纵向）— 全系统地基，最该先要** —— demo 用合成数据并刻意埋了背离/集中度异常。未来必须接 **ERP 或采购系统**，这是唯一不可替代的内部数据。
- 找谁：**采购部**（确认有哪些字段、口径）+ **IT/系统维护方**（开 §14.2 接口或定期导出）
- 最小集：物料、供应商、期次、单价、数量、金额；`freq`/`supplierShare` 能给更好，给不了可降级
- 过渡方案：先要一份历史 Excel 导出（⬜ 人工台账）跑通，再谈接口（🟥）

**③ 原材料行情（横向）** —— demo 联网抓铜价/CCL/金价公开快照。未来要"实时"得买数据源（🟨 如钢联/Wind/交易所行情）或让采购部维护一张材料价格台账（⬜）。优先级中，公开数据短期够用。

**④ 同行对标均价（横向）** —— demo 用公开财报/研报估算。未来精确化需 🟨 Wind/同花顺等付费库，或证券/战略部门的现成研报数据。优先级中低，主要服务"面子对比"。

**⑤ LLM 解读/建议** —— demo 预生成保证零翻车。未来切实时只需把 `llmClient` 指向你已有的 **Dify** 或 Claude API（§14.7），你自己就能搞定，不依赖他人。

**⑥ 预警推送出口** —— demo 仅本地弹窗。未来要真正"提醒有权管理人"需 🟥 接 **企业微信/OA** 推送接口（§14.5）。
- 找谁：**IT**（开推送接口）+ **行政/HR**（把"采购部负责人/分管副总"映射到真实账号）

**⑦ 有权管理人名单** —— demo 写死。未来从 HR 组织架构取，决定每类异常推给谁。与 ⑥ 配套，找 HR/行政。

**⑧ 阈值/规则配置** —— demo 硬编码（环比10%、背离8%、集中度70% 等）。未来应让采购负责人**确认并可调**这些阈值（⬜ 配置项），因为阈值定多严直接决定预警多不多，这是业务决策不是技术决策。

### 15.3 推进优先级建议（你一个人推的顺序）
1. **先要 ② 采购明细的历史 Excel** —— 一份导出就能把纵向从"模拟"变"真实"，性价比最高、最能堵"数据哪来的"质疑。
2. **③ 原材料行情** —— 公开抓取先顶着，不急着买数据源。
3. **⑤ LLM 接 Dify** —— 你自己可控，随时升级。
4. **⑥⑦ 推送 + 名单** —— 等 Stage1 过了、要"真发提醒"时再拉 IT/HR，正好是踢给业务部门的 Stage2 内容。
5. **④ 同行精确数据、① 主数据接口** —— 最后再说，demo 阶段不必纠结。
